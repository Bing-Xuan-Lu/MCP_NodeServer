# 使用者全域指引

---

## 檔案讀取效率

- **Read 工具預設可讀 2000 行**，大部分檔案一次讀完即可，不要分段讀取（每次 tool call 都有 token 開銷）
- 若只需查特定函式或關鍵字，優先用 **Grep** 定向搜尋，避免讀取整個檔案
- 超過 2000 行的大檔案才需要分段，且分段步幅應盡量大（500–2000 行）

## 檔案編輯效率

- 修改少量內容用 **Edit**（只傳差異），不要用 **Write** 覆寫整檔
- 同一區塊的多處修改盡量**合併成一次 Edit**，減少 tool call 來回
- Edit 前必須先 Read 過該檔案，不要憑記憶猜測內容（猜錯 = 重試 = 浪費）

---

## 腳本處理工具優先順序（Windows Git Bash 環境）

- **Edit 工具** → 檔案小幅修改，最快，零開銷
- **Node.js via Bash** → 文字/JSON 批次處理（Git Bash 裡真實可用）
- **Docker PHP** → PHP 執行/語法驗證（`php -l`）
- **Python 不適用**：Windows 上 `python3` 是 Microsoft Store stub，Git Bash 呼叫會開啟 Store 而非執行

## Skill MD 寫入規範（跨專案適用）

新增或修改 `{MCP_ROOT}\Skills\commands\` 下的**公開 Skill MD** 時，禁止寫入：
- 客戶實際網址、域名、專案名稱、模組名稱、資料表名稱
- `localhost` 帶 port（`localhost:{port}` ✗ → `localhost` ✓），port 是專案特定的
- AxShare 密碼等憑證

範例一律使用 `{ProjectFolder}`、`{module}`、`example.com`、`localhost` 等佔位符。

**例外**：`_internal/` 資料夾下的內部 Skill 可自由寫入專案實際資訊。

> 特別注意：從其他專案工作時新增 Skill，腦中全是該專案的 URL 和模組名，很容易不自覺寫進 MD 檔。**完成後必須掃一遍確認無洩漏。**

## MCP_Server 跨專案持續改進

無論當前在哪個專案工作，若對話中發現以下情況，**主動提議**修正 `{MCP_ROOT}`：

**觸發條件**（符合任一即提議）：
- MCP 工具回傳錯誤或行為不符預期（bug 修正機會）
- 發現現有工具缺少常用參數或功能（工具增強機會）
- 重複手動操作可被自動化為新工具或 Skill（新增機會）
- Skill 執行流程有缺陷、步驟遺漏、或產出品質不佳（Skill 改進機會）
- 開發過程中歸納出可復用的最佳實踐（Skill 模式化機會）

**提議格式**：
```
🔧 發現 MCP 改進機會：
  - 類型：{Bug 修正 / 工具增強 / 新增工具 / Skill 改進 / 新增 Skill}
  - 目標：[具體描述]
  - 影響範圍：[哪些專案/工作流受益]
  要現在處理嗎？
```

**執行原則**：
- 使用者同意後才修改 `{MCP_ROOT}` 下的檔案
- 修改後遵循 CLAUDE.md 的文件同步規範（tools 改動 → 更新 CLAUDE.md / README.md / dashboard.html）
- Skill 改動後執行部署核對清單
- 若改動量大，建議先在當前任務完成後再處理

## 對話經驗自動存檔

當對話中發生以下情況時，**主動提議**將經驗存入 auto-memory（不自動寫入，等使用者同意）：

**觸發條件**（符合任一即提議）：
- 解決了一個花費多輪對話才搞定的問題（踩坑 → 找到根因 → 修復）
- 發現 Skill 執行時的設計缺陷並修正（如範圍膨脹、假設錯誤）
- 使用者糾正了 Claude 的錯誤判斷（代表既有認知有誤）
- 建立了新的工作流程或慣例（如「部署前一定要先跑 X」）
- 發現了跨專案可復用的技術方案

**提議格式**：
```
💡 這次的經驗值得存入記憶嗎？
  - [一句話摘要]
  - 分類：{踩坑紀錄 / 架構決策 / 工作流慣例 / Skill 改進}
  - 存入：MEMORY.md 或 memory/{topic}.md
```

**寫入原則**：
- 使用者同意後才寫入，拒絕就不存
- 檢查現有 memory 檔案避免重複，已有相同主題則更新而非新增
- MEMORY.md 保持精簡（< 200 行），詳細內容放主題檔並從 MEMORY.md 連結
- 記錄「結論」而非「過程」— 寫「X 情況下應該用 Y 方法」，不寫「我們先試了 A 失敗，再試 B...」

**持久化方式選擇**（以節省 token 為首要原則）：
- **預設用 Memory**：存結論性知識（規則、決策、feedback），按需讀取，token 消耗極小
- **例外用 Session 存檔**：僅在任務明確未完成 + 狀態複雜到無法用幾行摘要 + 使用者表示下次接著做，三者同時成立時才存 session。session 檔只存未完成事項 + 關鍵決策 + 阻塞點，不存完整對話重現
