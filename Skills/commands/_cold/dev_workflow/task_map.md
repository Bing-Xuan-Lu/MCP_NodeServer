# /task_map — 從使用者旅程產出專案全景心智模型

你是專案架構分析師，從「終端使用者會做什麼」出發，而非從「系統有哪些模組」出發，產出一張涵蓋前台使用者旅程、後台管理流程、模組間資料流向的全景地圖。

## 背景

傳統的 codemap 只記錄檔案結構（骨架），缺少「血肉」— 使用者旅程、模組間的資料如何流動、哪條路徑是業務核心。Task Map 補足這個缺口，成為 `/sprint_plan`、`/e2e_golden_path`、`/sadd` 的導航基礎。

---

## 使用者輸入

$ARGUMENTS

格式：`{ProjectFolder} [規格來源]`

- `{ProjectFolder}`：專案名稱（對應 `D:\Project\{ProjectFolder}\`）
- `[規格來源]`（可省略）：規格書路徑、AxShare URL、或 spec_index.md

---

## 需要的資訊

若使用者未提供以下資訊，請主動詢問：

| 參數 | 說明 | 預設值 | 範例 |
|------|------|--------|------|
| ProjectFolder | 專案資料夾名稱 | — | `{ProjectFolder}` |
| 前台網址 | 前台可存取的測試網址 | — | `http://localhost/{ProjectFolder}/` |
| 後台網址 | 後台管理介面 URL | 前台 + `/adminControl/` | — |
| 規格來源 | 規格書路徑或 AxShare URL | 無（從程式碼推導） | `reports/spec_index.md` |
| 業務領域 | 一句話描述這個系統做什麼 | — | `B2C 電商網站` |

## 執行步驟

### 步驟 1：三軌並行資訊收集

同時從三個來源收集資訊：

**A. 前台使用者視角（Playwright）**

```
browser_navigate({前台網址})
browser_snapshot()
→ 蒐集主導覽列、頁尾連結、所有可見的使用者入口
→ 識別：訪客能做什麼？會員能做什麼？
→ 截圖首頁存 screenshots/task_map_home.png
```

**B. 後台管理視角（Playwright 或目錄掃描）**

```
list_files_batch → 掃描 adminControl/ 下所有模組
→ 或 browser_navigate({後台網址}) → 蒐集後台選單結構
→ 識別：管理者操作哪些資料？
```

**C. DB Schema 視角**

```
get_db_schema_batch → 取得所有資料表
→ 識別外鍵關係（誰引用誰）
→ 找出中心表（被最多表引用的 = 核心實體）
```

---

### 步驟 2：建立使用者旅程

從「一個外部顧客第一次造訪到完成核心目標」拆解旅程步驟：

**拆解原則**：
- 每一步 = 使用者的**一個意圖**（「我想看商品」「我想加到購物車」）
- 不是系統的內部動作（不是「呼叫 addToCart API」）
- 以使用者的**感知**為單位（「頁面上看到什麼變了」）

**典型旅程範本**（依業務領域調整）：

```text
B2C 電商：
  訪客 → 瀏覽首頁 → 搜尋/分類 → 商品詳情 → 加入購物車
  → 註冊/登入 → 結帳 → 填寫配送 → 選擇付款 → 完成訂單
  → 查看訂單 → 退貨/客服

B2B 報價系統：
  業務 → 登入 → 建立客戶 → 建立報價單 → 選擇品項
  → 調整價格 → 送審 → 主管核准 → 轉訂單 → 出貨

內部管理系統：
  員工 → 登入 → 填寫表單 → 提交 → 主管審核 → 結案
```

對每一步，記錄：
1. 使用者看到的頁面 / UI
2. 使用者的操作（點什麼、填什麼）
3. 系統的回應（頁面變化、訊息）
4. 背後涉及的模組 + DB 表 + API

---

### 步驟 3：標示跨模組資料流

對旅程中每一步，追蹤資料如何在模組間流動：

```text
格式：
[使用者操作] → {前台頁面} → {API/Controller} → {DB 表} → {影響的其他模組}

範例：
[點擊加入購物車] → product_detail.php → ajax/cart.php → cart_items 表 → 購物車圖示數字更新
[送出訂單] → checkout.php → ajax/order.php → orders + order_items 表 → 庫存扣減(products) + 金流請求(payments)
```

---

### 步驟 4：識別黃金路徑與風險點

**黃金路徑（Golden Path）**：
- 從旅程中選出**一條最核心的端到端路徑**
- 標準：如果這條路斷了，整個系統就沒有商業價值
- 這條路徑將直接輸出給 `/e2e_golden_path` 使用

**風險點標示**：

| 標記 | 意義 |
|------|------|
| `[跨模組]` | 此步驟的資料跨越 2 個以上模組，改壞一個會影響其他 |
| `[第三方]` | 依賴外部 API（金流/物流/SMS），不可控 |
| `[狀態機]` | 涉及狀態轉換邏輯（訂單狀態、審核流程），容易出 bug |
| `[計算]` | 涉及金額/庫存/折扣計算，需要精確測試 |
| `[權限]` | 不同角色看到/能做的不同 |

## 注意事項

- 旅程從**使用者的意圖**出發，不從系統模組出發 — 先問「使用者想做什麼」，再問「系統怎麼做」
- 一個旅程步驟 = 使用者的一個意圖，不是系統的一個 API 呼叫
- 沒有前台的純後台系統，旅程改為「管理者的工作流程」
- 若規格書和實際程式碼有出入，以**實際程式碼行為**為準（task_map 記錄的是「現在是什麼」）
- Task Map 是快照，專案架構變動後需重新產出（呼叫 `/task_map` 覆寫）
