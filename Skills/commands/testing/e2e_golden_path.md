# /e2e_golden_path — 輕量黃金路徑煙霧測試

你是 E2E 煙霧測試員，用最快的速度走一遍系統的核心業務路徑，只驗證「主幹能不能跑通」，不做全站稽核。5 分鐘內給出 PASS/FAIL。

## 背景

`/project_qc` 是全站深度稽核（2-3 小時），適合驗收。但日常開發中每次改 code 後需要的是快速回歸 — 確認最核心的那條路沒有斷掉。黃金路徑 = 如果這條路斷了，整個系統就沒有商業價值的那條使用者旅程。

---

## 使用者輸入

$ARGUMENTS

- `{ProjectFolder}`（必要）：專案名稱
- `define`（可選）：只定義黃金路徑，不執行測試
- `run`（可選，預設）：執行已定義的黃金路徑測試
- `{golden_path.md 路徑}`（可選）：指定既有的路徑定義檔

## 執行步驟

### 步驟 1：載入或定義黃金路徑

**若已有定義檔**（`reports/golden_path.md` 或使用者指定路徑）：
- 讀取定義檔，跳至步驟 3

**若有 task_map.md**：
- 讀取 `reports/task_map.md` 的「黃金路徑」段落
- 轉為可執行的步驟定義

**若都沒有**：
- 依業務類型，從預設範本建立黃金路徑

**預設範本：**

```text
B2C 電商：
  1. [前台] 首頁載入 → 驗：頁面狀態 200，有商品顯示
  2. [前台] 點擊任一商品 → 驗：商品詳情頁載入，有價格
  3. [前台] 加入購物車 → 驗：購物車數量 +1
  4. [前台] 進入購物車 → 驗：商品項目顯示正確
  5. [前台] 註冊新帳號（或登入） → 驗：登入狀態
  6. [前台] 結帳填表送出 → 驗：訂單成功頁面
  7. [DB] 驗證 orders 表有新記錄
  8. [後台] 登入 → 訂單列表 → 驗：剛建立的訂單出現

B2B 報價系統：
  1. [後台] 登入 → 驗：Dashboard 載入
  2. [後台] 建立客戶 → 驗：客戶列表出現新紀錄
  3. [後台] 建立報價單 → 選擇品項 → 驗：金額計算正確
  4. [後台] 送審 → 驗：狀態變更
  5. [DB] 驗證報價單資料

內部管理系統：
  1. [前台/後台] 登入 → 驗：首頁載入
  2. [操作] 建立核心資料（表單） → 驗：列表出現
  3. [操作] 編輯資料 → 驗：修改已儲存
  4. [操作] 提交/審核流程 → 驗：狀態正確
  5. [DB] 驗證資料完整性
```

### 步驟 2：產出黃金路徑定義檔

儲存至 `{ProjectFolder}/reports/golden_path.md`：

```markdown
# {ProjectFolder} — 黃金路徑定義

**業務類型：** {類型}
**預估執行時間：** {N} 分鐘
**最後更新：** {date}

## 路徑步驟

| # | 環境 | 使用者意圖 | 操作 | 驗證條件（PASS 標準） |
|---|------|-----------|------|---------------------|
| 1 | 前台 | {意圖} | {具體操作} | {具體可驗證的條件} |
| 2 | 前台 | {意圖} | {具體操作} | {具體可驗證的條件} |
| ...| DB  | DB 驗證 | execute_sql | {SQL + 預期結果} |
| N | 後台 | {意圖} | {具體操作} | {具體可驗證的條件} |

## 測試資料

| 用途 | 欄位 | 值 |
|------|------|-----|
| 註冊帳號 | Email | `golden_test_{timestamp}@test.com` |
| 註冊帳號 | 密碼 | `Test1234!` |
| ... | ... | ... |

## 清理規則

測試完成後清理以下測試資料：
- DELETE FROM {table} WHERE {condition}
```

> 若模式為 `define`，到此步驟結束，展示定義檔並詢問使用者確認。

### 步驟 3：執行黃金路徑測試

**執行原則**：

- **快速優先**：每步驟最多等 10 秒，逾時即 FAIL
- **不做深度檢查**：只驗「能不能跑通」，不驗 CSS 精確度、不驗所有欄位
- **失敗即停**：任一步驟 FAIL，記錄失敗點，不繼續後續步驟（因為路徑是串聯的）
- **截圖每一步**：每步操作後截圖，存 `screenshots/golden_path/step_{N}.png`

**前後台遞棒**（核心特色）：

前台操作完成後，切換到後台驗證資料是否正確反映：
```text
[前台] 使用者下單 → [DB] 確認 orders 有記錄 → [後台] 訂單列表有新訂單
```

這是 `/project_qc` Phase E 的精簡版 — 只走一遍黃金路徑，不做全站遞棒。

### 步驟 5：清理測試資料

依 golden_path.md 的「清理規則」，刪除本次測試產生的資料：

```
execute_sql_batch → 依序執行清理 SQL
→ 確認清理完成（查詢確認 0 筆）
```

> 若使用者指定 `--no-cleanup`，跳過此步驟。

## 常見錯誤

| 症狀 | 原因 | 解法 |
|------|------|------|
| 步驟 1 就 FAIL（首頁 500） | 環境沒起來 | 確認 Apache/Nginx + DB 運作中 |
| 註冊步驟 FAIL（帳號已存在） | 上次測試沒清理 | 手動清理或用 timestamp 帳號 |
| 後台看不到前台建的訂單 | 帳號權限不夠 | 確認後台帳號有管理員權限 |
| 加入購物車無反應 | JS 錯誤 | 檢查 browser console 錯誤訊息 |

## 注意事項

- 黃金路徑是**最短的核心路徑**，不是全站測試 — 控制在 8-12 步以內
- 失敗即停：路徑是串聯的，步驟 3 失敗代表步驟 4 以後都無法執行
- 測試帳號用 `golden_test_{timestamp}@test.com` 避免衝突
- 每次 `run` 都是獨立的（建立新測試資料 → 測試 → 清理）
- 與 `/project_qc` 的關係：golden_path 是煙霧測試（5 分鐘），project_qc 是全站稽核（2-3 小時）
- 建議開發流程：改 code → `/e2e_golden_path run` → PASS → commit → 階段性 `/project_qc`
