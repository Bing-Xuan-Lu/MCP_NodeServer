# E2E 業務流程測試協議

> 本檔由 `/project_qc` 主 Skill 在 Phase E-0 / Phase E 階段讀取。

## Phase E-0 — 業務流程測試矩陣

> **解決組合爆炸問題：先發現維度，再規劃代表性案例，不窮舉**

派遣 UI/UX 稽核員（使用 Playwright）探索網站，產出測試矩陣：

1. 爬取商品/服務頁面，發現組合維度：
   - 規格維度（顏色/尺寸/款式/材質...）
   - 數量邊界（最小購買量、庫存上限）
   - 可用優惠類型（折扣碼/紅利點數/滿額折/限時活動...）
   - 付款方式（信用卡/ATM/超商...）
   - 特殊條件（會員等級/地區限制/商品組合...）

2. 用等價分割法與邊界值分析，選出代表性案例：
   每個維度取「正常值 + 邊界值 + 異常值」各 1 個
   不同維度的組合取最高風險的前 N 組（N ≤ 10）

### 矩陣範例（購買流程）

| 案例 | 商品規格 | 數量 | 優惠 | 付款 | 驗證重點 |
|------|---------|------|------|------|---------|
| T01 | 單規格 | 1 | 無 | 信用卡 | 基本流程 |
| T02 | 多規格 | 最大量 | 無 | ATM | 庫存上限 + 規格對應 |
| T03 | 任意 | 1 | 折扣碼+點數疊加 | 任意 | 金額計算 |
| T04 | 任意 | 1 | 過期優惠碼 | 任意 | 異常阻擋 |
| T05 | 庫存=0 | 1 | 無 | 任意 | 無法下單提示 |

### 矩陣範例（訂單生命週期）

| 案例 | 流程階段 | 前台操作 | 後台操作 | 驗證重點 |
|------|---------|---------|---------|---------|
| L01 | 下單→待付款 | 前台送出訂單 | 後台列表出現 | 訂單狀態正確、金額一致 |
| L02 | 待付款→付款完成 | 前台模擬付款 | 後台狀態更新 | 付款紀錄、金額對帳 |
| L03 | 待處理→列印訂單 | — | 後台列印操作 | 承辦人欄位記錄、PDF/列印頁正確 |
| L04 | 待出貨→已出貨 | — | 後台出貨操作 | 出貨單、物流編號、前台狀態更新 |
| L05 | 已出貨→已完成 | 前台確認收貨 | 後台狀態更新 | 訂單結案 |
| L06 | 取消訂單 | 前台申請取消 | 後台審核取消 | 退款流程、庫存回補 |
| L07 | 退貨流程 | 前台申請退貨 | 後台審核退貨 | 退貨狀態轉換、退款計算 |
| L08 | 再次訂購 | 前台「再次訂購」按鈕 | — | 購物車帶入舊規格、品號後綴遞增 |

**再次訂購（L08）特別驗證**：
- 商品的品號/編號後綴是否正確帶入
- 舊訂單的規格選項是否完整帶入新購物車
- 帶入後是否可修改規格再重新計算

### 矩陣範例（邊界與清理）

| 案例 | 流程階段 | 操作 | 驗證重點 |
|------|---------|------|---------|
| B01 | 登出清除 | 加項目到購物車 → 登出 → 查看購物車 | 購物車應為空 |
| B02 | 庫存阻擋 | 庫存設0 → 加入購物車 | 應被拒絕 |
| B03 | 金額計算 | 多項目+運費+折抵 → 結帳 | 應付金額 = Σ小計 - 折抵 + 運費 |
| B04 | 免運門檻 | 加入超過免運金額項目 | 運費=0、免運提示消失 |
| B05 | 紅利折抵 | 輸入紅利 → 驗合計 | 合計 = 原價 - 紅利 |

### 矩陣範例（訂單子 Tab 逐一進入）

**常見漏測：訂單管理只測了「全部」tab，子 tab 完全沒進去。必須逐一驗證。**

| 案例 | Tab 名稱 | URL 路徑 | 驗證重點 |
|------|---------|---------|---------|
| T-TAB01 | {status_1} | {module}/{status}/list.php | 列表可載入、欄位與規格一致 |
| T-TAB02 | {status_2} | {module}/{status}/list.php | 有匯出按鈕（依規格） |
| T-TAB03 | {status_3} | {module}/{status}/list.php | 批次操作只處理勾選項 |
| T-TAB04 | {status_N} | {module}/{status}/list.php | 欄位與規格一致 |

> 以上為範例結構，實際 Tab 名稱與數量依專案規格書定義。

每個 Tab 除了「能打開」外，還要對照規格書驗：欄位清單、按鈕清單、文字。

輸出 `reports/test_matrix.md`，**請使用者確認矩陣後再繼續**。

## Phase E — 遞棒協議

依確認後的測試矩陣，每個案例執行遞棒協議：

### UI/UX 稽核員（Playwright）

依矩陣案例執行使用者操作（下單/申請/送出表單），截圖每個關鍵節點。
操作完成後建立 `reports/handoff.json`：

```json
{
  "case": "T03",
  "flow": "購物流程",
  "action": "前台下單完成",
  "payload": {
    "order_id": "前台顯示的訂單編號",
    "expected_amount": "計算值",
    "product_id": "...",
    "sku_id": "...",
    "quantity": "...",
    "discount_applied": "折扣碼名稱或點數金額"
  }
}
```

### 邏輯稽核員（讀取 handoff.json，不開瀏覽器）

`execute_sql` → 驗 DB 對應資料：
- 訂單是否存在（orders 表）
- 金額是否與 expected_amount 一致
- 庫存是否正確扣減
- 優惠/點數是否正確計算
- 後台管理頁面可用 `send_http_request` 驗 API 回傳

結果追加至 `reports/e2e_results.md`。

依序執行所有矩陣案例（T01 → T02 → ... → TN），每個案例完成後再跑下一個。

### 測試數據證據要求（不可省略）

每個矩陣案例執行完畢後，必須記錄**具體數據**：

```markdown
### 案例 T01 — {流程名稱}

**操作步驟與截圖**：
1. 開啟頁面 → 截圖：`screenshots/e_t01_01_page.png`
2. 選擇規格：{選項A}={值1}、{選項B}={值2}...
   → 截圖：`screenshots/e_t01_02_selected.png`
   → 頁面顯示單價：${金額}
3. 數量輸入：{N} → 小計顯示：${計算值}
4. 加入購物車 → 購物車顯示 {N} 項，金額 ${計算值}
5. 結帳 → 填寫資訊 → 送出
6. 完成頁 → 編號：{實際訂單編號}，總金額：${實際金額}

**DB 驗證（邏輯稽核員）**：
- {orders 表}：{主鍵}={實際值}, total={實際值}, status={實際值} ✅/❌
- {明細表}：qty={實際值}, price={實際值} ✅/❌
- 庫存變化：{欄位} 從 {舊值} → {新值} ✅/❌

**結論**：PASS/NG（附 N 張截圖 + N 筆 SQL 查詢結果）
```

**禁止的記錄方式**：
- 「已完成流程，操作成功」（無具體數據）
- 「金額正確」（正確是多少？）
- 「DB 驗證通過」（查了什麼？值是什麼？）

### Phase E 完成後自驗迴圈

全部案例跑完後，**必須讀回 `reports/e2e_results.md`** 執行自驗：

1. `read_files_batch` 讀取 `reports/e2e_results.md`
2. 逐案例檢查：截圖路徑？具體數值？SQL 結果？
3. `[UNVERIFIED]` > 0 → 回頭補做（上限 2 輪）
4. 仍無法取得 → 標記 `[SKIP-無法取得證據]` 並附原因

## Phase E-special — 特殊功能測試前置確認

若業務流程涉及**發信**或**支付**，在進入 Phase E 前需先完成以下前置確認。

### 發信功能

在步驟 0 詢問資訊時加問：

```text
是否有發信功能需要測試？（如：下單確認信、密碼重設信）
若有，請確認以下任一條件已滿足：
  1. 已設定測試 SMTP（如 Mailtrap、MailHog），請提供 SMTP Inbox 查看方式
  2. 系統有 mail_log 資料表記錄寄信（提供表名）
  3. 可接受僅驗證「觸發點正常、無 PHP 錯誤」，不驗收信件內容
```

| 方式 | 驗證方法 |
|------|---------|
| Mailtrap/MailHog | `send_http_request` GET API 取得收件箱；或 Playwright 開 Web UI 截圖 |
| mail_log 資料表 | `execute_sql` SELECT 最新一筆，比對 to/subject/status |
| 僅驗觸發點 | `tail_log` 確認無 PHP Fatal，HTTP 200，無 JS 錯誤 |

handoff.json 補充：`"email": { "expected_to": "...", "expected_subject": "...", "verify_method": "mail_log | mailtrap | trigger_only" }`

### 支付功能

在步驟 0 詢問資訊時加問：

```text
是否有支付功能需要測試？
若有，請工程師先完成以下設定：
  □ 支付閘道已切換為「沙盒/測試模式」（Sandbox Key）
  □ Webhook callback URL 已設定為測試環境網址
  □ 測試信用卡號已備妥
  □ 若需等待 callback，請告知最長等待時間（秒）
```

**支付測試策略**：

UI/UX 稽核員：操作結帳頁至付款 → 輸入測試卡號 → 截圖 → 等待跳轉 → 截圖結果 → 寫入 handoff.json

邏輯稽核員：驗 orders.payment_status → 驗 payment_logs callback 原始資料 → 驗庫存扣減

**Webhook Callback 等待機制**：
1. UI/UX 付款後，先截圖「等待確認」頁面
2. 邏輯稽核員輪詢 DB（每 5 秒，最多 60 秒）
3. 60 秒未到達 → 標記 [PENDING]，通知使用者確認 Webhook 設定
4. 到達後 → 補跑驗證，更新 handoff.json 與 e2e_results.md
