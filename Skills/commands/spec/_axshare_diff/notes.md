# /axshare_diff — 注意事項

> 此檔案由 `/axshare_diff` 在完成比對後讀取確認。

---

## 通用規則

- 先完成兩端擷取再比對，不要邊抓邊猜
- 以規格書為基準，「缺少」= 規格有但實作沒有
- **標記 ❓ 待確認前必須交叉驗證**（避免把已有答案的項目標為待確認）：
  1. Grep 程式碼 — 該功能是否已實作
  2. 查 DB/DBML — 該欄位是否已存在
  3. 查同份報告其他段落 — 是否前文已回答
  4. 查 `spec/` 目錄其他文件 — 是否已有決議
  5. 查 MEMORY.md — 使用者是否在先前對話中已確認
  若以上任一找到答案 → 直接寫入決議，不標 ❓
- 此 Skill 僅做分析報告，不修改任何程式碼（ALTER TABLE SQL 只列出不執行）

## 全站報告模式

- 不讀取 PHP 原始碼，只比對「規格書畫面」vs「網站渲染結果」
- 三種報告格式都要產出（總覽表格 / 逐頁比對 / 待辦清單）

## 單一單元模式

- **嚴禁擅自擴大比對範圍**：只比對使用者指定的模組。發現其他模組差異僅在末尾一句話列出
- **可以**讀取 DB schema (DBML) 和 Model 類別，用於 DB 影響評估
- 結果直接整合進現有報告檔（不另產獨立報告），保留原有「開發者備註」
- 更新報告後同步更新「統計摘要」和「時間戳」，比對統計表嵌入 TODO 章節內

## 模組校閱模式

- 適用於子頁面多（>5 頁）的複雜模組，報告輸出為 `spec/{module}_diff_report.md`
- 本地索引不含 layout 結構，搜尋區/列表欄位仍需 Playwright 截圖確認
- DB 狀態檢查是必要步驟 — 空表會導致 UI 功能被隱藏
- P1/P2/P3 優先級由「功能完整性影響」決定，非由「修改工作量」決定
- 每個子頁面的比對結果獨立成一個 section，方便逐頁修正參照

## 日期註記掃描模式

- **不需要**測試網站和登入（步驟 2、3 全部跳過），只掃描 AxShare 規格書
- 整合報告由主 agent 統一執行，確保章節定位和格式一致
- 「純前台 UI」變更（僅 CSS/排版/文案）不整合到後台差異報告，僅在統計中記錄
- 整合後必須更新報告時間戳，格式：`{目標日期}規格書更新整合: {今日日期} (變更摘要)`

## 後續清理步驟（所有模式通用）

比對完成後，**必須**執行以下兩項檢查並向使用者報告：

1. **樣式合規掃描**：對比對範圍內的 list.php 執行 `/dbox3_style_lint`，掃描按鈕樣式是否統一。將結果附加到報告的待辦清單中。

2. **孤兒檔案/表掃描**：檢查該模組目錄下是否有無引用的 PHP 檔案、規格書中不存在的頁面、規格書中不存在但 DB 有的表。結果列出並**主動詢問使用者是否要刪除**。
