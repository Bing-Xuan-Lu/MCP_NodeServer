# /sprint_plan — 依賴層快速開發計畫（AI 極速開發模式）

你是 AI 極速開發規劃師，根據規格書或現有目錄結構，分析模組依賴關係，輸出可執行的分層開發計畫（`sprint_plan.md`）。同層模組平行執行，跨層串行，兼顧速度與 token 控制。

---

## 使用者輸入

$ARGUMENTS

格式：`{ProjectFolder} [規格來源]`

- `{ProjectFolder}`：專案名稱（對應 `D:\Project\{ProjectFolder}\`）
- `[規格來源]`（可省略）：規格書路徑或 spec 索引；省略時從目錄結構推導

## 執行步驟

### 步驟 1：收集模組清單

**若有規格來源：**

- 讀取規格書，提取所有功能模組與頁面清單

**若無規格來源：**

- `list_files({ProjectFolder}/)` 掃描目錄結構，每個子資料夾視為一個模組

**若已有 sprint_plan.md（更新模式）：**

- 讀取現有計畫，保留狀態為「完成」的模組，重新規劃剩餘部分

---

### 步驟 2：依賴分析，建立 Tier

分析每個模組的依賴關係，依以下規則分層：

| Tier | 條件 |
| ---- | ---- |
| Tier 1 | 無任何依賴（DB 設定、Auth、基礎設定） |
| Tier 2 | 只依賴 Tier 1 的模組 |
| Tier 3 | 依賴 Tier 2 的模組 |
| Tier N | 依此類推 |

**依賴判斷依據（優先順序）：**

1. **DB 外鍵**（最可靠）：呼叫 `get_db_schema_batch` 取得所有資料表，找出 `FOREIGN KEY` 關係，有外鍵依賴的模組排在被依賴者之後
2. **業務流程依賴**（外鍵抓不到的）：使用者旅程中 A 步驟必須在 B 之前完成（如「結帳前必須有購物車」），即使 DB 無外鍵也要排序。若有 `reports/task_map.md`，從黃金路徑直接讀取依賴順序
3. **規格書功能說明**：A 模組的頁面需要顯示 B 模組的資料 → A 依賴 B
4. **目錄結構推導**（無 DB/規格時的備用）：從現有 controller / model 檔案引用關係推導

---

### 步驟 2.5：任務評估維度（同 Tier 內排序依據）

同一 Tier 內的模組不一定等價。對每個模組填寫以下五個維度，決定**執行順序**和**回報節奏**：

| 維度 | 評分 | 判斷方式 | 影響 |
| ---- | ---- | -------- | ---- |
| 陌生度 | 高/中/低 | 團隊/AI 過去有沒有做過類似模組？（Memory 中有無相關經驗） | 高陌生 → 先做（儘早暴露未知） |
| 過去經驗 | 有/無 | 同類型專案是否做過？（如：做過 3 次電商購物車） | 有經驗 → 估時可壓縮 20% |
| 技術複雜度 | 高/中/低 | 涉及狀態機、計算邏輯、第三方串接的數量 | 高複雜 → 拆更小的 demo-able 任務 |
| 規模與影響 | 大/中/小 | 改壞後影響多少其他模組？（查 task_map.md 的跨模組資料流） | 大影響 → 完成後立即跑 `/e2e_golden_path` |
| 回報頻率 | 每任務/每日/每 Tier | 使用者需要多頻繁看到進度？ | 高陌生 + 大影響 → 每任務回報 |

**回報頻率自動判定規則**：

| 條件 | 回報頻率 |
| ---- | -------- |
| 陌生度=高 或 影響=大 | 每任務完成後回報 + 跑 `/e2e_golden_path` |
| 陌生度=中 且 影響=中 | 每日回報 |
| 陌生度=低 且 影響=小 | 每 Tier 完成後統一回報 |

**評估結果填入 sprint_plan.md 的模組表格**（新增欄位，見步驟 5 輸出格式）。

---

### 步驟 3：判斷每個模組的測試策略

| 模組類型 | 判斷條件 | 策略 |
| -------- | -------- | ---- |
| 標準 CRUD | 列表 / 新增 / 編輯 / 刪除，無複雜計算 | `/sadd` → `/php_crud_test` |
| 複雜邏輯 | 規格書含計算公式、狀態轉換、多條件權限 | `/tdd` → `/sadd` → `/php_crud_test` |

**複雜邏輯模組的特徵（若符合以下任一項則標記為 `/tdd` 前置）：**

- 規格書中有明確計算公式（金額、折扣、庫存扣減）
- 狀態機（訂單狀態轉換：待付款 → 已付款 → 出貨中）
- 多角色權限邏輯（管理員 / 業務 / 客服 可執行的操作不同）
- 跨模組聯動副作用（修改 A 會自動觸發 B 的變化）

---

### 步驟 4：Token 估算

以「對話輪數」估算每個模組的 AI 執行成本：

| Task 類型 | 估算（輪） | 說明 |
| --------- | --------- | ---- |
| DB Migration（ALTER/CREATE） | 0.5 輪 | AI 一次產出，幾乎不需修正 |
| CRUD 模組（後台） | 1–2 輪 | 用 generator 或樣板，快速產出 |
| 前台頁面（簡單：blog/news/FAQ） | 1 輪 | 改樣式 + 微調邏輯 |
| 前台頁面（中等：表單/列表/篩選） | 2–3 輪 | 需規格確認 + UI 調整 |
| 前台頁面（複雜：訂單/結帳/報價） | 4–6 輪 | 多條件邏輯 + 測試 + debug |
| AJAX 端點（簡單 GET/POST） | 0.5–1 輪 | 模式固定 |
| AJAX 端點（複雜：金額計算/狀態轉換） | 2–3 輪 | 需邏輯驗證 |
| JS 互動（POPUP/Tab/HOVER） | 1–2 輪 | |
| CSS/RWD | 1–2 輪 | 需反覆截圖比對 |
| 第三方串接（金流/SMS/API） | 2–4 輪 | 文件研讀 + sandbox 測試 |
| 整合測試（Playwright E2E） | 1–2 輪 | AI 寫測試腳本 + 執行 |
| 規格書校對 | 1 輪 | 每模組至少 1 輪 |
| /tdd 前置（複雜邏輯模組） | +1 輪 | 先寫測試，再交給 /sadd |

估算後加 10% buffer。

**不可壓縮的瓶頸項目（估時維持原值）：**

- 規格書歧義 → 需人工確認
- 第三方 API sandbox 測試 → 等待回應
- CSS 像素級精修 → 需反覆截圖
- Playwright 偵錯 → session 干擾、timing 問題
- 跨模組副作用 → 需回歸測試

## 注意事項

- **更新模式**：`sprint_plan.md` 已存在時，保留「完成」模組，只重新規劃剩餘 Tier
- 每個模組的完成條件以**規格書校對**為主，不以 git commit 為依據
- `/tdd` 只用於規格書含計算邏輯、狀態機、多角色權限的模組；標準 CRUD 跳過，直接 `/sadd`
- 若使用 `/logic_trace` 已產出模組邏輯文件，複雜模組可壓縮 1–2 輪（AI 有完整上下文）
- 規格書歧義項目一律列入「規格歧義待確認清單」，不讓 AI 自行猜測
- 不估算「設計」工項（UI 設計由設計師負責）
