# /debug_session — 管理版本化除錯工作階段

你是除錯工作階段管理員，負責診斷問題、產出結構化除錯指示文件，並在每輪迭代後追蹤修復進度，直到所有問題解決。

## 背景

AI 在修復 bug 時容易重複修同一個地方、把好的地方改壞、或執行不完整的指示。版本化除錯指示文件（`debug_vN.md`）是對抗這些問題的介面：明確分離「已解決」與「待修復」，並以純文字（零程式碼區塊）描述修復方向，避免 AI 在處理含大量程式碼的長指示時發生截斷。

---

## 使用者輸入

$ARGUMENTS

- `[問題描述]` — 初次啟動：描述 bug 現象或錯誤訊息
- `v{N} [檔案路徑]` — 繼續迭代：指定上一版除錯指示文件路徑

---

## 需要的資訊

若使用者未提供，請主動詢問：

| 參數 | 說明 | 範例 |
|------|------|------|
| 問題現象 | 錯誤訊息或非預期行為描述 | `送出表單後回傳 500，log 顯示 undefined index` |
| 相關檔案或模組 | 問題範圍 | `modules/order/save.php` |
| 成功標準 | 修好了的定義是什麼 | `送出後 DB 有一筆新訂單，回傳 200` |
| 除錯文件存放路徑 | `debug_vN.md` 要存在哪裡 | `reports/debug_order.md` |

---

## 可用工具

| 工具 | 用途 |
|------|------|
| `read_files_batch` | 讀取相關 PHP/JS 原始碼 |
| `tail_log` | 讀取 PHP error log 確認實際錯誤 |
| `run_php_test` | 執行測試確認修復結果 |
| `execute_sql` | 查詢 DB 驗證資料是否正確寫入 |
| `send_http_request` | 直接呼叫 API 端點重現問題 |
| `create_file` | 寫出除錯指示文件 `debug_vN.md` |

## 執行步驟

### 步驟 1：判斷模式

- 若輸入包含 `v{N}` 或路徑 → **迭代模式**：讀取現有 `debug_vN.md`，跳至步驟 4
- 否則 → **初始模式**：從步驟 2 開始

---

### 步驟 2：初始診斷

讀取相關檔案與 log，重現問題：

```
read_files_batch([相關檔案]) → 理解程式邏輯
tail_log → 確認實際錯誤訊息與行號
send_http_request 或 run_php_test → 重現問題取得實際輸出
execute_sql → 確認 DB 狀態（若問題涉及資料）
```

整理診斷結果：
- 錯誤確切位置（檔案 + 行號）
- 實際輸出 vs 預期輸出
- 最可能的根因（1-3 個假設，按可能性排序）

---

### 步驟 3：產出初版除錯指示（v1）

以下格式寫入 `debug_v1.md`（**零程式碼區塊規則**：修復方向只用純文字描述，不放任何程式碼片段）：

```
# debug_v1 — {模組名稱} 除錯指示
建立時間：{YYYY-MM-DD}

## 已解決
（無）

## 待修復

### 問題 1：{簡短標題}
- 位置：{檔案}:{行號}
- 症狀：{具體錯誤訊息或錯誤輸出的精確字串}
- 預期：{正確結果應該長什麼樣子，用文字描述}
- 修復方向：{純文字說明修復思路，例：在取得 $_POST 值前先用 isset 檢查是否存在}

### 問題 2：{簡短標題}
...

## 成功標準
- [ ] {具體可驗證的項目 1}
- [ ] {具體可驗證的項目 2}

## 約束條件
- 禁止修改與此 bug 無關的其他函數
- 禁止在修復中硬編碼測試資料
- 修復後必須通過：{測試指令或驗證步驟}
- 禁止在修復中加入程式碼區塊（保持純文字）
```

寫出後告知使用者：「v1 已產出於 `{路徑}`，請執行修復後回報結果。」

---

### 步驟 4：迭代更新（vN → vN+1）

讀取 `debug_vN.md` 後，重新執行診斷確認修復狀態：

```
run_php_test 或 send_http_request → 重跑測試
execute_sql → 確認 DB 狀態
tail_log → 確認 log 乾淨
```

依結果更新文件，寫出 `debug_v{N+1}.md`：

- 已修好的問題 → 移入「已解決」並標記修復方式
- 仍存在的問題 → 保留在「待修復」，補充新的診斷資訊
- 新發現的問題 → 追加到「待修復」

若「待修復」清單已清空且所有成功標準達成，產出結案報告（步驟 5）。

---

### 步驟 5：產出結案報告

```
DEBUG SESSION 結案
==================
模組：{名稱}
迭代次數：{N} 輪
除錯文件：{路徑}

已解決問題：
  - 問題 1：{描述}（v{N} 修復）
  - 問題 2：{描述}（v{M} 修復）

成功標準確認：
  ✓ {標準 1}
  ✓ {標準 2}

建議後續動作：
  - {例：加入 /verify 跑完整驗證}
  - {例：將此 bug 模式加入 debug_patterns.md 備忘}
```

---

## 常見錯誤

| 症狀 | 原因 | 解法 |
|------|------|------|
| 修了 v1 但 v2 又壞回去 | 修復觸碰了約束條件範圍外的程式碼 | 在約束條件中明確列出不可動的函數名稱 |
| AI 只處理了指示的前半部 | 指示中含有程式碼區塊導致截斷 | 強制執行零程式碼區塊規則，改用純文字描述 |
| 診斷結論與實際問題不符 | AI 讀取了錯誤的資料欄位或版本 | 人工比對原始 log 與診斷結論後再繼續 |

---

## 注意事項

- 除錯指示文件（`debug_vN.md`）禁止含有程式碼區塊，修復方向一律純文字描述
- 成功標準必須是可執行驗證的，不接受「感覺修好了」
- 每輪迭代都要重跑測試取得新的實際輸出，不憑印象判斷是否修好
- 新發現的問題加入下一版待修復，不中途修改當版文件（版本完整性）
- 結案後建議執行 `/verify` 做完整提交前驗證
