## 三次找不到就問

搜尋資料時，最多嘗試 3 次不同方向，若仍找不到立即問使用者。

**Why:** 避免浪費大量 tool call 在錯誤方向上，使用者通常知道正確位置。

**How to apply:** 搜尋檔案、DB 欄位、設定值等任何資料，嘗試 3 次仍無結果 → 立即停下，詢問「找不到 X，請確認資料來源位置」。

---

## 文件沒有就直接進系統

規格文件、memory、git log 都找不到某功能最新狀態時，不要繼續翻文件，直接進系統確認。

**Why:** 文件可能過時或未更新，直接看實際系統狀態最快最準。

**How to apply:** 搜尋文件三次無解 → 改為：(1) 直接問使用者，或 (2) 用 Playwright / curl 進系統實測確認現況。

## RAG vs Grep 搜尋選擇

| 場景 | 工具 |
|------|------|
| 知道具體函式名/變數名 | **Grep** |
| 不確定功能在哪個檔案（語意搜尋）| **RAG** |
| 後台 CRUD 頁面 | 直接 **Read** 對應模組的 list.php |
| RAG 搜尋結果不相關（distance > 0.5）| 改用 **Grep** |

RAG 索引 paths 必須精確指向**專案內子目錄**（帶專案名前綴），不可指向母資料夾，否則會把同層其他專案混入 collection。
