## 規格書驅動測試（不是 checklist 驅動）

QC 測試必須以規格書為驅動，不能只測事先列好的 checklist。

**Why:** 預列的 checklist 一定會漏。應該像 PM 一樣：打開規格書 → 進系統 → 逐頁逐欄比對。

**How to apply:**
1. 打開規格書索引，逐頁進入系統對應頁面
2. 比對規格書每個欄位、按鈕、行為、排版
3. 不符的截圖（規格書截圖 + 實際截圖）記錄
4. **規格書外額外測試維度**（spec 不會明確寫但必測）：後台→前台連動、分頁（塞超過上限）、表單防呆、RWD 三斷點

## ❓待確認前必須交叉驗證

產出「❓待確認」前，必須先確認該問題是否已有答案。

**Why:** 已有答案的項目被標 ❓ 浪費使用者逐項確認的時間（曾發生 40 項中 7 項其實已有答案）。

**How to apply:** 標記 ❓ 前依序檢查：
1. **Grep 程式碼** — 該功能是否已實作
2. **查 DB** — 該欄位是否已存在
3. **查同份文件其他段落** — 是否前文已回答
4. **查 spec/ 其他文件** — 是否已有決議
5. **查 MEMORY.md** — 使用者是否在先前對話中已回答過

以上任一找到答案 → 直接寫入決議，不標 ❓。

## 規格比對後主動清理孤兒檔案

axshare_diff 或規格書比對完成後，必須主動掃描並詢問是否刪除多餘檔案。

**Why:** 多餘的 PHP 檔案留在專案中會誤導規格理解（孤兒檔案讓人以為功能還在使用）。

**How to apply:**
1. 比對完成後，用 Grep 掃描模組目錄下每個 PHP 檔案的引用數
2. 引用數 = 0 的檔案列出標記「可刪除」
3. 規格書中不存在但目錄中有的頁面標記「規格外」
4. 主動向使用者報告並詢問是否刪除
5. 注意：ECPay/金流回調 URL 即使無內部引用也可能被外部系統呼叫，需特別標記

## AxShare 規格書無法透過 HTTP 直接解析

`GET axshare.com` 頁面返回 JS redirect to Axure Cloud，邏輯稽核員用 `send_http_request` 無法解析頁面內容。

**Why:** 邏輯稽核員嘗試 HTTP 取得規格書內容，全部返回 JS redirect 而非實際 HTML，無任何規格書文字可供比對。

**How to apply:**
1. 邏輯稽核員 Phase D：spec 來源必須是**本地 spec_index.md**（由 `/axshare_spec_index` 預先產生），不能是 AxShare URL
2. 若本地 spec_index.md 不存在 → 立即標記 [SKIP-本地快照不存在] → **不嘗試 HTTP 爬 AxShare**
3. AxShare 線上規格書的頁面比對（截圖）→ 由 **UI/UX 稽核員（Playwright）** 負責，Playwright 可真正打開網頁
4. 大型 QC 前建議先跑 `/axshare_spec_index` 建立本地快照，邏輯稽核員才能做文字比對

## 校稿單格式（比對表）

| # | 項目 | 規格書 | 設計稿 | 實際 | 結果 |
|---|------|--------|--------|------|:----:|

- 前台頁面：規格書 + 設計稿 + 實際（三欄比對）
- 後台頁面：規格書 + 實際（兩欄，後台無設計稿）
- 每個問題必須附截圖佐證（無佐證 = 問題無效）
- 按模組分組（`## 【模組名稱】`），每個問題獨立編號
